### 5.3 迭代改进的循环

> **破除焦虑：** "我知道评估重要，但评估完了怎么改？改完怎么确认改对了？"→一个五步循环，从头走到尾。

> **一句话心法：** 每一轮迭代都是一次完整的贝叶斯更新——定义标准、测量基线、分析错误、针对性优化、持续监控。

---

评估不是终点，是起点。评估完不改，等于白评。改完不验证，等于白改。验证完不监控，等于白忙。

你需要一个循环——不是一个"试试看"的循环，而是一个有结构、有纪律的循环。这个循环有五步，每一步都对应贝叶斯更新中的一个关键环节：

**第一步：定义"好"的标准。**

这是整个循环的地基。如果你不知道"好"长什么样，后面所有步骤都是空中楼阁。

"好"的标准不能是模糊的——"感觉还行""大致过得去"不叫标准。标准必须是可观察、可量化、可再生成的。比如：

- 合同生成应用："每个条款必须包含主体、条件、违约后果，且违约后果的金额计算必须正确"
- 客服回复应用："回复必须在三句话以内解决用户问题，且不得包含任何AI无法核实的承诺"
- 代码生成应用："生成的代码必须通过预设的单元测试，且不得使用废弃的API"

定义标准就是在给AI立"验收规矩"——第3章说的"交付规格书"。没有验收规矩，你连"违约"都发现不了。

**第二步：测量基线。**

标准定好了，跑一轮用例集，看看当前方案在标准上的表现——这就是你的基线。

基线就是你当前的先验概率。你会得到一组数据：事实准确率、逻辑一致率、格式合规率、关键信息完整率……这些数字不是用来焦虑的——"天啊才60%！"——而是用来定位的："好，起点在60%，接下来往哪里使劲？"

没有基线，你永远不知道改进是真实的还是感觉上的。很多团队改了一轮Prompt，觉得"好多了"——但基线数据一比，改进不到5%，完全是随机波动。测了才知道。

**第三步：分析错误。**

基线跑出来了，接下来是最有技术含量的一步：错误分类。

不是所有错误都一样——这一点怎么强调都不为过。错误至少分五类：

**事实错误**：AI输出了错误的事实。比如把"诺贝尔物理学奖"说成"诺贝尔和平奖"，或者金额算错、日期搞混。这类错误最致命——因为用户会直接质疑AI的可信度。

**逻辑错误**：事实可能是对的，但推理链条断裂或矛盾。比如"因为A，所以B"——但A和B之间没有因果关系。这类错误最隐蔽——因为表面看起来通顺，仔细想才发现站不住脚。

**格式错误**：内容可能没问题，但输出格式不合规。比如要求输出JSON但给了Markdown，要求列表但给了散文。这类错误最容易被发现，也最容易修复。

**遗漏**：该说的没说。比如合同里必须有"违约条款"但AI没生成，用户问题问了三件事但AI只回答了两件。这类错误最容易被忽略——因为你可能也不知道AI漏了什么。

**过度发挥**：不该说的说了。比如用户问"这个条款什么意思"，AI不仅解释了，还自作主张修改了条款。或者用户要一个简要回复，AI给了一篇论文。这类错误最狡猾——因为看起来AI"很努力"，实际上它在越界。

每个错误类型对应不同的调整策略：

- 事实错误 → 加RAG或知识库，让AI查得到事实
- 逻辑错误 → 加思维链或验证步骤，让AI一步步推理
- 格式错误 → 强化Prompt中的格式约束，加示例
- 遗漏 → 加检查清单，让AI过一遍再输出
- 过度发挥 → 加边界约束，明确"不要做什么"

错误分类就是你在算调整因子——不同类型的错误，调整力度和方向完全不同。一刀切地"改Prompt"解决不了五类不同的问题。

**第四步：针对性优化。**

分析完错误，改的方向就清楚了。但改的时候有一个原则：**一次只改一个变量。**

如果你同时改了Prompt的三处、换了模型、还加了RAG——结果变好了，你不知道是哪个改动起了作用。结果变差了，你更不知道该撤哪个。这跟科学实验是一个道理：控制变量，一个一个来。

具体的优化动作，根据错误类型来：

- 事实错误多 → 加RAG，让AI在回答前先检索知识库
- 逻辑错误多 → 加思维链，让AI先输出推理过程再给结论
- 格式错误多 → 在Prompt里加格式示例和约束条件
- 遗漏多 → 加一个"检查步骤"——AI输出后先自查清单，确认没遗漏再返回
- 过度发挥多 → 在Prompt里加明确的"不要"清单

每一轮优化只针对一类主要错误。改完之后，再跑一轮用例集，看这类错误是不是减少了——这就是下一轮的"收集证据"。

**第五步：持续监控。**

改进完了，达标了——恭喜。但别急着开香槟。

模型的API可能升级、用户的输入模式可能变化、知识库可能过时——任何一个外部变动都可能让你的后验概率悄悄退化。你需要持续监控：定期跑用例集，对比基线，看看有没有新的退步。

监控的关键是**不信任记忆，只信任数据**。你觉得"最近好像没问题"，不算数。用例集跑出来的数据才算数。法官不会因为"最近没出过错案"就不审查——他得每案必审。你也一样。

---

五步循环走完一轮，你完成了一次完整的贝叶斯更新：

1. 定义标准 = 确定衡量"好"的尺子
2. 测量基线 = 量化当前的先验概率
3. 分析错误 = 算出调整因子
4. 针对性优化 = 按调整因子修正行为
5. 持续监控 = 确保后验不退化

一轮循环的结果，是更准的后验概率和更好的行为。而下一轮的先验，就是这一轮的后验。你不是从零开始——你永远站在上一轮的认知基础上往前走。

这就像调试一段复杂的代码：你不会只看一次输出就说"搞定了"——你会跑测试、看报错、改代码、再跑测试。每跑一轮，你对这段代码的理解就深一层。AI应用也一样，只不过你调试的不是代码，而是"行为-概率-证据"的关系。

> **概率工具箱**
>
> 迭代改进五步循环：
> 1. **定义标准**——"好"必须可观察、可量化、可再生成
> 2. **测量基线**——当前先验概率是多少，用数据说
> 3. **分析错误**——分类五维度：事实、逻辑、格式、遗漏、过度发挥
> 4. **针对性优化**——一次只改一个变量，每类错误有对应的调整策略
> 5. **持续监控**——不信任记忆，只信任数据
>
> 一轮循环 = 一次完整贝叶斯更新 = 正确概率又逼近一步。
