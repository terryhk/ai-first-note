### 4.3.1 延伸阅读：MCP 与 CLI 的争论——标准化协议 vs 五十年的沉淀

2024年底Anthropic推出MCP时，行业为之振奋——一套标准协议，让AI像USB-C一样即插即用地连接任何工具。但到了2026年初，风向变了。Vercel CEO在X上直言"CLIs are the de-facto MCPs for agents"；Y Combinator掌舵人Garry Tan公开站队CLI；中文社区甚至有人喊出"MCP已死"。

这不是意气之争。这场争论的核心是：**AI连接外部世界，到底需要一个新的标准化协议，还是直接用已经存在了五十年的命令行？** 两个方向都有坚实的论据和真实的数据，值得深入理解。

#### CLI派的论据：为什么说CLI才是Agent的"母语"

一位字节跳动工程师做了一组有趣的统计：他个人Agent过去一个月执行的工具调用中，Bash命令占了78%，MCP调用只占12%，剩下10%是文件读写。而那12%的MCP调用里，大部分可以用CLI替代。

这不是个例。他进一步做了基准测试，直接对比同一批任务在CLI和MCP下的表现：

| 指标 | CLI | MCP |
|------|-----|-----|
| 单次调用成本 | 基准 | 高10-32倍 |
| 任务完成可靠率 | 100% | 72% |
| 初始化开销 | 几乎为零 | Schema加载（数万tokens） |
| 错误恢复 | 重新执行命令即可 | 重新建立连接+加载Schema |

数据背后有三层原因：

**第一层：Token经济学。** LLM的训练语料中包含了数十亿行CLI命令和输出——Stack Overflow、GitHub Issues、man pages、Shell脚本。模型"天生就会"使用CLI，像母语一样自然。而MCP的JSON-RPC协议和tool schema是2024年才定义的新规范，模型需要在有限的学习数据中"死记硬背"怎么用，多了一层毫无必要的格式转换。

微软Playwright团队的亲身体验最有说服力：他们先做了Playwright MCP Server，然后发现一个15步的浏览器自动化会话需要消耗约114,000 tokens。于是2026年初他们推出了Playwright CLI，同样的15步任务只需约27,000 tokens——**同一个团队，同一款产品，差距4倍以上。** 该团队的README里写道："Modern coding agents increasingly favor CLI-based workflows over MCP because CLI invocations are more token-efficient."

而Anthropic自己的Claude Code也走了同样的路：早期用MCP做代码执行消耗约150,000 tokens，后来改成原生代码执行只需约2,000 tokens——**降了75倍。MCP的亲爹绕开了MCP。**

**第二层：可组合性与可调试性。** CLI天生可组合：你可以通过管道把输出传给jq、与grep链式组合、重定向到文件。AI执行 `gh pr view 123` 时出错了？你可以在终端运行同一行命令，精确复现问题。MCP的工具调用存在于LLM的对话内部——出问题了你得去翻JSON日志，而不是简单自己跑一下命令。正如一位开发者所说："调试工作不该依赖协议解码器。"

**第三层：成熟的安全模型和认证机制。** CLI背后是Unix进程模型——每个命令独立运行，有明确的权限边界，出问题也只影响这一个命令。认证机制（`gh auth login`、`aws configure`）经过了数十年实战检验。而MCP的安全模型被Simon Willison（Django联合创始人）指出存在"致命三连"：**工具投毒**（Server在tool description里藏恶意指令）、**地毯抽拉**（Server悄悄更新行为）、**影子工具**（恶意Server注册同名工具覆盖原有行为）。所有Server共享Agent的上下文和权限——一个恶意Server就能看到所有交互数据。

#### MCP派的回应：连接层的价值不可替代

CLI派的批评很有力，但MCP支持者的回应同样有道理。

**核心反驳：CLI的前提是工具必须已安装。** 你能用 `gh` 的前提是已经 `brew install gh` 过了。Agent需要连接几十个不同的远程服务——数据库、消息队列、第三方API——每个都装一个本地CLI不现实。MCP的服务发现能力在这里有真实价值：它提供标准化的方式描述远程服务的能力和接口。

**Schema开销有优化的空间。** 一个GitHub MCP Server加载tool schema要55,000 tokens，这确实是当前的问题。但这不是协议本身的缺陷——Schema可以按需加载、可以缓存、可以用更紧凑的格式。MCP还很年轻（2024年底才发布），性能优化远没到头。

**适合特定场景。** 多租户环境需要统一的授权管理、企业合规需要结构化的审计日志、沙箱环境里CLI本身就没法用——这些场景下MCP比CLI更合适。Playwright团队自己也说：有文件系统权限的Agent用CLI，沙箱环境用MCP。

#### 业界共识：互补分工，各司其职

到2026年中，行业内逐渐形成的共识是："CLI做执行层，MCP做连接层"——CLI负责Agent的日常操作（高效、省token），MCP负责远程服务的发现和接入（标准化、有认证），两者协同工作。

具体来说：

- **CLI适合**：本地化、确定性、高速操作——文件操作、代码执行、系统管理。这些AI天生就会，不需要额外协议层。
- **MCP适合**：远程服务连接、多租户权限管理、企业合规审计——需要标准化接口定义和统一认证管理的场景。

一个典型的混合架构示例：

> 行业简报Agent的工作流：
> 1. [MCP] 调用新闻聚合API发现本周热文列表
> 2. [CLI] `curl` 获取每篇文章的原始内容
> 3. [CLI] 用本地脚本做文本清洗和提取
> 4. [MCP] 调用摘要服务API对每篇文章做结构化摘要
> 5. [CLI] 用 `pandoc` 将结果转成Markdown格式
> 6. [MCP] 调用邮件服务API发送简报

#### 给本书读者的建议

作为AI使用者而非基础设施开发者，你不需要在CLI和MCP之间"选边站"。你需要理解的是：

- **两者不是替代关系。** MCP没有"杀死"CLI，CLI也没有"杀死"MCP。它们解决的是不同抽象层级的问题。
- **关注Token成本。** 如果你的Agent任务消耗tokens异常高，检查是否过度依赖MCP——很多操作用CLI就能做，成本低一个数量级。
- **安全优先。** 在生产环境中使用第三方MCP Server时，务必评估其安全风险。相比之下，命令行工具经过了半个世纪的攻防对抗检验。
- **工具越好，越不需要特殊协议。** 如果一个服务连基本的CLI都没有，优先要求他们提供CLI——而不是为此搭建MCP Server。正如一位工程师所说："你不需要为AI专门搭建复杂的接口，只要API和CLI做得好，AI自己就能学会怎么用。"
