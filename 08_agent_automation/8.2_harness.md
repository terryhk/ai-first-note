# 8.2 Harness：Agent的"运行时容器"

有了Skill，Agent就有能力了。但能力还不够——你还需要一个东西来管理Agent的整个运行过程。这个东西叫Harness（运行时容器）。

再用手机的类比：Skill是App，那Harness是什么？是iOS或Android本身。你平时几乎感觉不到操作系统的存在——你点开微信就直接用，不会想"操作系统帮我分配了内存、调度了CPU、管理了网络连接"。但如果没有操作系统，所有App都跑不起来：微信没法联网，音乐播放器没法发声，地图没法定位。操作系统在背后默默做的一切——生命周期管理、资源调度、权限控制、安全隔离——就是Harness为Agent做的事。

具体来说，Harness管理五件事：

**生命周期。** Agent什么时候启动、什么时候暂停、什么时候关闭——就像操作系统管理App的启动和退出。一个Agent如果跑着跑着卡住了，Harness负责检测并重启它，而不是让你等半天以为它还在干活。

**工具调度。** Agent要用搜索工具了，Harness帮它找到对应的Skill并调用；Agent要发邮件了，Harness帮它连上邮件API。就像操作系统帮你把App的请求转发到对应的硬件——你要拍照，系统把请求转给摄像头；你要定位，系统把请求转给GPS。

**上下文管理。** 多轮对话里的历史消息、长文档的分块缓存、跨任务的记忆保留——都由Harness打理。你不会希望你的Agent跑着跑着就把之前做的事忘了，就像你不会希望手机切了一下后台微信就把聊天记录丢了。

**安全沙箱。** Agent能做什么、不能做什么——边界由Harness执行。你的Agent能帮你搜新闻，但你不希望它擅自帮你下单买东西。安全沙箱就像手机的权限管理：App要访问你的通讯录，系统会弹窗问你"允许吗？"；App没有你的同意，读不到你的通讯录。Agent也一样，Harness会给它划好能动的圈——出了圈就不行。

**模型路由。** 同一个Agent流程可以切换不同底层模型。简单的问题用便宜的轻量模型，复杂的推理用强力的旗舰模型——就像手机根据任务轻重自动切换CPU的大核和小核，省电又高效。

Harness这个概念最近为什么这么火？因为一个趋势越来越明显：**模型能力趋同，真正差异在"怎么用模型"。** DeepSeek、千问、GLM这些国产模型的差距在缩小，GPT、Claude、Gemini之间的代际差也在收窄。当模型本身的差异越来越小，竞争力就转移到了应用层——谁的工具编排更高效、工作流更智能、安全控制更可靠，谁就赢。这些事情，都是Harness的活。

判断原则：**你不需要学会写Harness，但你需要理解它的存在。** 理解Harness能帮你做三件事：第一，知道Agent为什么能自动做事——背后有Harness在调度一切，不是魔法；第二，遇到"Agent为什么不工作"时，知道排查方向——可能不是模型的问题，是Harness的配置问题；第三，理解Skill和Harness的关系——Skill是Harness装载的"弹药"，没有Harness这把枪，弹药打了不出去。

我自己用的Harness是基于OpenClaw搭建的。OpenClaw把生命周期、工具调度、上下文管理、安全沙箱、模型路由都封装好了，我只需要关注两件事：装什么Skill、配什么流程。这就像用iPhone——你不需要理解iOS内核，你只需要在App Store里选App、排小组件。
