# 本章小结

### 📊 概率工具箱

回到全书的核心框架：行为 × 概率 = 结果。Agent自动化的本质，是让你在"行为"侧用SOP + Skill + Harness替代手工操作，从而把AI输出"成功率"的概率拉高。

| 场景 | 无方法 P | 有方法 P | 方法 |
|------|---------|----------|------|
| 手工做行业简报 | ≈60% | — | 人工筛选+摘要，但人容易疲劳漏信息 |
| AI单次对话做简报 | ≈30% | — | 一轮对话内完成，缺多步迭代，信息不完整 |
| Agent + Skill + Harness | — | ≈80%+ | SOP驱动多步ReAct，关键点人在环审批 |
| Agent完全放手（无人在环） | — | ≈50% | 跑偏无纠正，错误累积 |

数据说明两件事：第一，Agent + SOP + 人在环的组合，比手工操作的成功率更高（80% vs 60%）——因为Agent不会疲劳、不会跳行、不会因为赶时间而偷工减料。第二，完全放手是陷阱——没有人在环的Agent，成功率只有50%，还不如手工做。检查点就是概率校准的锚——每过一个检查点，你就在做一次贝叶斯更新，把Agent的输出从"可能对"校准到"确认对"。

### 边际成本与机会成本分析

Agent自动化的经济账很清晰：**一次配置SOP的投入，换来后续每次执行的边际成本趋近于零。** 这和第7章周报模板的逻辑一样，但层级更高——模板只固化输出格式，Agent固化的是整个执行流程。

来看看行业简报的经济账：

| | 手工做 | Agent + 人在环 |
|------|--------|---------------|
| 初始投入 | 0小时 | ~4小时（设计SOP + 配置Skill + 试跑迭代） |
| 每周耗时 | 2-3小时 | 20-30分钟审校 |
| 月均节省 | — | 7-10小时 |
| 年均节省 | — | 85-120小时 |

但Agent自动化隐藏着两个经济学风险——如果你只看"省了多少时间"而不看这两个风险，迟早会踩坑。

**第一个风险：委托-代理问题。**

你把任务委托给了Agent（代理人），但你和Agent的利益并不完全一致。Agent会追求"完成任务"——但它不一定追求"高质量完成"。如果你没有设定好的Harness（运行时容器）来约束行为、没有人在环来检查产出，Agent就会像偷懒的员工一样——交了差就算完事，质量嘛，过得去就行。

《经济学思维方式》告诉我们，解决委托-代理问题的核心是"激励相容"——你要设计SOP和检查点，让Agent的利益和你对齐。具体来说：你在SOP里写的每一条约束，就是在告诉Agent"这个维度我有要求，你别糊弄"；你设的每一个检查点，就是在告诉Agent"我会看的，你认真做"。没有约束的Agent，就像没有KPI的员工——它不是不想做好，是没有人告诉它什么叫"好"。

**第二个风险：公地悲剧。**

当一个Agent被多人共用，且没有明确的资源配额和权限边界时，每个使用者都会倾向于"多占用资源"——更多的搜索次数、更长的上下文、更频繁的API调用——最终系统被拖垮。这不是理论推演，这是我已经遇到过的事：我们的团队简报Agent，三个人同时用的时候搜索配额被打满了，Agent跑一半搜索API报限流错误，简报出到一半停住了。

公地悲剧的解法和经济学教科书里写的一样：**界定产权 + 设定额度和权限。** Harness中的安全沙箱和权限控制不是"可选的高级功能"，而是"必须的基础设施"——就像小区的停车位不是先到先得，而是每户有固定编号。没有产权边界，公地必然被过度使用。

更深一层的机会成本：你省下的时间，如果不用来做更有价值的事（比如做判断、做决策、做学习），那省下来的时间就等于白省了。自动化释放的时间是"机会"，不是"结果"——你得把它用在比自动化任务更有价值的地方，这个账才算赢。

### 📋 可复用模板

下面三个模板，覆盖了Agent任务设计最常见的场景。直接复制，填入你自己的内容即可。

#### 模板一：Agent任务SOP设计模板

```
任务名称：[填写]
目标：[一句话描述Agent要达成的最终结果]

ReAct循环定义：
第1轮：
- 思考：[Agent应该想什么]
- 行动：[调用哪个Skill/工具]
- 观察：[期望得到什么结果]

第2轮：
- 思考：[Agent应该想什么]
- 行动：[调用哪个Skill/工具]
- 观察：[期望得到什么结果]

...（按需增加轮次）

检查点设定：
| 步骤 | 检查点等级 | 检查内容 |
|------|-----------|---------|
| [步骤名] | 自动放行/事后抽查/事前审批 | [看什么] |

完成标准：[怎样算做完了]
失败处理：[Agent跑偏了怎么办——重跑/降级为手工/报警]
```

#### 模板二：行业动态简报Agent工作流

```
任务名称：行业动态简报
目标：每周一9:00前，生成指定行业的周动态简报并发送至邮箱

ReAct循环定义：
第1轮（搜索）：
- 思考："我需要搜索本周[行业名]动态，关键词：[关键词列表]"
- 行动：调用web搜索Skill
- 观察：收到N条搜索结果

第2轮（过滤）：
- 思考："筛掉广告、旧闻、无关内容，保留最近7天的有效新闻"
- 行动：调用文本过滤Skill（排除含"赞助/推广/广告"字样的结果；只保留7天内发布的内容）
- 观察：剩下M条有效新闻

第3轮（摘要）：
- 思考："为每条新闻写50字以内摘要"
- 行动：调用文本摘要Skill
- 观察：M条摘要已生成

第4轮（分类排版）：
- 思考："按技术/商业/政策分类，按简报模板排版"
- 行动：调用模板生成Skill
- 观察：简报邮件初稿已生成

第5轮（发送审批）：
- 思考："初稿完成，提交审批"
- 行动：调用审批/通知Skill
- 观察：等待审批结果

检查点设定：
| 步骤 | 检查点等级 | 检查内容 |
|------|-----------|---------|
| 搜索 | 自动放行 | — |
| 过滤 | 事后抽查 | 随机抽2条确认是否为有效新闻 |
| 摘要 | 事后抽查 | 随机抽2条对照原文确认准确性 |
| 分类排版 | 事前审批 | 格式、分类、信息完整性 |
| 发送 | 事前审批 | 最终确认 |

完成标准：简报邮件经审批后已发送
失败处理：搜索无结果→扩大关键词范围；摘要不准确→修复摘要SOP Prompt；审批不通过→修改后重新排版
```

#### 模板三：竞品监控Agent工作流

```
任务名称：竞品动态监控
目标：每天早上8:00前，监控指定竞品的最新动态，生成变动摘要

ReAct循环定义：
第1轮（搜索）：
- 思考："搜索[竞品名]的最新动态，关键词：[竞品名]+发布/更新/融资/合作"
- 行动：调用web搜索Skill
- 观察：收到N条搜索结果

第2轮（去重+过滤）：
- 思考："与昨日记录比对，筛掉已收录的旧信息，保留新增内容"
- 行动：调用去重Skill（对比历史记录数据库）
- 观察：剩下M条新增信息

第3轮（分类+摘要）：
- 思考："按产品更新/市场动作/人事变动/融资新闻分类并摘要"
- 行动：调用文本分类+摘要Skill
- 观察：分类摘要已生成

第4轮（生成报告）：
- 思考："按竞品监控模板格式化，标注变化程度（大/中/小）"
- 行动：调用模板生成Skill
- 观察：监控报告初稿已生成

第5轮（通知）：
- 思考："有新增变动→即时通知；无变动→日报汇总"
- 行动：调用通知Skill（IM/邮件）
- 观察：通知已发送

检查点设定：
| 步骤 | 检查点等级 | 检查内容 |
|------|-----------|---------|
| 搜索 | 自动放行 | — |
| 去重 | 自动放行 | — |
| 分类摘要 | 事后抽查 | 抽1-2条确认分类和摘要准确性 |
| 生成报告 | 事前审批 | 变动程度标注是否合理 |
| 通知 | 自动放行（大变动除外） | 大变动需事前审批后通知 |

完成标准：监控报告已发送；有大变动时经审批后即时通知
失败处理：搜索受限→切换备用信息源；无新增→发送"今日无变动"即可
```

---

这三个模板的用法都一样：复制到你的Agent配置里，替换方括号里的内容，试跑几轮，根据输出调整Prompt和检查点。刚开始你可能需要三四轮迭代才能让Agent稳定在80%以上的成功率——前几小时是投入期。但一旦调好了，后面的每一周，你只需要审校几十分钟。从"每次两三小时"到"每次二三十分钟"，差额就是你赢得的思考时间。

Agent不是让你"什么都不做"，而是让你"只做最有价值的事"——在关键节点做判断，而不是在每一个执行步骤上亲力亲为。这就是半人马协作的终极形态：马跑全程，人在岔路指方向。
